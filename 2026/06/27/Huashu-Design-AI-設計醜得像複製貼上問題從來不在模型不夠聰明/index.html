Huashu Design — AI 設計醜得像複製貼上，問題從來不在模型不夠聰明

發表於2026-06-27|更新於2026-06-27|AI 開發工具
|總字數:2.2k|閱讀時間:6分鐘|瀏覽量:

AI 設計做出來的東西很醜，但醜的原因不是它笨。

你大概看過這個畫面。叫 AI 設計一個 App，它回你紫色漸變的背景、幾個 emoji 當 icon、圓角卡片再配一條彩色的左邊框。乾淨、能看，但沒有一個真的品牌會把這種東西放上線。設計圈給它取了名字，叫 AI slop——AI 量產出來的視覺垃圾。

直覺的歸因是「模型品味差」。可是同一個模型，你叫它寫一段並發處理的程式碼，它能寫得有模有樣；你叫它幫你 debug 一個 race condition，它能講出個道理。寫程式的時候它一點都不笨，換到設計就崩。同一顆腦袋，差別到底在哪？

差別在於：寫程式有大量明文規則，設計沒有。

醜不是審美問題，是規則缺席問題

一個資深設計師看到「紫漸變加 emoji 圖示」會皺眉，但你問他為什麼不行，他多半只會說「就是不對」「沒質感」。這就是麻煩的地方——設計師的判斷力大多是隱性知識，存在他的手感裡，沒有被寫成一條條可以照做的規則。模型學不到沒被寫下來的東西。它看過的訓練資料裡，紫漸變加 emoji 出現了一百萬次，那它當然覺得這就是「設計」。

把這件事想透，AI slop 就不是模型缺陷，是知識缺陷。沒人把「資深設計師為什麼皺眉」翻譯成機器能執行的指令，模型只好回退到統計上最常見的那種長相。

花叔（GitHub ID alchaincyf）2026 年 4 月初開源的 Huashu Design，狠就狠在它正面去解這件事。它不是再給模型多塞幾句 prompt、多花一點 token 求它「設計得高級一點」，而是把資深設計師的判斷力整包編碼下來——一套 workflow、二十份 reference 文件、一批 starter components、再加上會自己跑的驗證腳本。發佈幾天就衝上 2.2k stars、360 多個 fork，是今年 AI 設計工具這塊最爆紅的開源專案之一。

它是一份 skill，不是一個產品

先把心智模型擺正，後面的設計才看得懂。

Huashu Design 不是 SaaS、不是 GUI 工具、也不是 Figma 替代品。它是一個 Claude Code skill。而 Claude Code 的 skill 講白了就是「一份 markdown 加一堆參考資產」，agent 讀完之後就照著裡面的指引工作。所以它本質上是一個方法論壓縮包。

核心是那份 SKILL.md，足足 56 KB 的超大指引；外圍是二十份 references/*.md，涵蓋投影片架構、動畫陷阱、設計風格庫、評審規範；再來是 assets/ 資料夾，裝著 iPhone 15 Pro 的裝置外框、Stage/Sprite 動畫引擎、背景音樂、音效庫；最後 scripts/ 裡放著一條匯出工具鏈。

裝起來也只有一行：

不用 npm install、不用 build pipeline、不用啟什麼 dev server。如果你用的是 Cursor、Codex 或其他相容的 agent，直接 git clone 丟進它的 skills 目錄也能跑——作者列出的支援平台有五個以上。跟 Anthropic 官方那個綁 Claude.ai、閉源付費的 Claude Design 比起來，Huashu 走的是 CLI 優先、跨 agent、開源而且個人免費的路線。

裝完之後你只要在對話框講人話：「我想做一個 SaaS 登入頁，給我三個風格方向比稿」「幫我做一份十頁的募資簡報」「做個 30 秒的動畫講神經網路怎麼運作」。剩下的它接手。

把「品味」拆成可以照做的步驟

這才是重點。一個工具能不能治本 AI slop，看的不是它會不會講漂亮話，是它有沒有把抽象的「好設計」拆成模型做得到的動作。Huashu Design 的心臟是一條九步工作流程，每一步都有明確任務和產出：先做事實驗證、理解需求、品牌資產協議、位置四問，然後停下來等你確認方向；接著交一版半成品、再停下來等你確認；確認完才填細節、做變體、跑驗證、匯出。

注意那兩個停下來的檢查點。它最聰明的設計，是不准 agent 一路衝到底。AI 設計工具最常見的悲劇，就是你下一句指令，它跑十分鐘，回來給你一坨成品，方向整個歪掉，全部重來。強制在關鍵節點停下來對焦，等於把「跑歪十分鐘」的成本砍掉。

裡頭最有設計師味道的一條硬規則，叫 5-10-2-8 品牌資產協議。如果你要做涉及真實品牌的設計，比方「幫 Stripe 做一份募資簡報」，它會強制你跑完：至少 5 輪搜尋、收集 10 個候選資產、實際下載而不是用文字描述代替、精選出 2 個品質 8 分（滿分 10）以上的，最後把規格寫死進一份 brand-spec.md，後面全部依它走。設計師常講「好的高保真稿是長在 context 上的，不是憑空畫在白紙上」——這條協議就是把「收集 context」這個動作強制化、而且可以驗證。

再來是反 AI slop 的明文黑名單，直接點名禁止：紫色漸變背景、emoji 當功能性圖示、圓角卡片配彩色左邊框、用 CSS 剪影冒充產品照。同時它強制偏好一些技術選型：text-wrap: pretty 讓文字自動平衡斷行、用 CSS Grid 而不是 flexbox 硬堆、挑精選的 serif 顯示字體來對抗「一律 Inter」的審美疲勞、色彩走 oklch 空間因為它比 RGB 更貼近人眼感知。你看，每一條都是把一個設計師「就是不對」的直覺，翻譯成了一句機器能照做的規則。

收尾還有一個我第一次在開源 skill 看到的東西：可編輯的 PPTX 匯出。作者自己寫了一個 37 KB 的 html2pptx.js，能把 HTML 轉成文字框還能編輯的簡報檔，不是點陣化的圖片。交付給同事後，他們能直接在 PowerPoint 裡改字。最後再讓 Playwright 自動把所有互動點一遍，檢查 console 有沒有噴錯。設計到這裡，已經不太像「請 AI 畫圖」，比較像「請一個照 SOP 辦事的初級設計師交件」。

它的限制，作者自己講得最清楚

工具吹得再響，誠實邊界才看得出成色。這點花叔處理得很乾脆：他自評這是「一個 80 分的 skill，不是 100 分的產品」。

它做不了複雜動畫——3D、物理模擬、粒子系統超出 Stage/Sprite 模型的能力。它也不是 Figma 替代品，沒有圖層級匯出、沒有協作模式、沒有 design system token 管理。沒有品牌資產墊底的空白畫布設計，作者自評只有 60 到 65 分。要做需要後端、需要 SEO 的生產級 Web App，他直接叫你去用另一個 frontend-design skill，不要硬凹。還有一個小細節很有意思：這個 repo 的 contributors 只有兩個，花叔本人加上 Claude（掛 co-author）。這大概是我看過對 AI 協作標註得最誠實的開源專案。

（以上整理自我知識庫裡對這個 repo 的筆記與其公開文件，部分能力我還沒每一項都自己跑過。）

真正的啟示：品味是一串還沒被寫下來的規則

繞回最開頭那個問題。AI 設計醜，從來不是因為模型不懂美，是因為「美」這件事一直躺在某些人的手感裡，沒被翻譯成它能執行的東西。Huashu Design 證明的事很簡單：你把資深設計師皺眉的那些瞬間，一條一條寫成硬規則和可驗證的流程，模型立刻就不 slop 了。

這背後其實只有一條線：當「生成」這個動作的成本被壓到趨近於零，按下生成鍵不再值錢，價值就整個往上游退——退到「誰能把模糊的好壞判斷，講成清楚到機器照得做的規格」。設計如此，寫程式其實也是，這幾年工程師的價值不也正從打字速度，退到把需求想清楚、切乾淨、講成機器能照做的樣子。

下次再嫌 AI 做的東西沒質感，先別急著怪它笨。問問自己：你心裡那把尺，有沒有哪一格是能寫下來、卻一直沒寫下來的？
