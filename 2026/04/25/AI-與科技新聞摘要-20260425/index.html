AI 與科技新聞摘要 - 2026/04/25

發表於2026-04-25|更新於2026-05-14|AI與科技新聞

300 萬。

這是 OpenAI Codex 的週活躍開發者數字，截至 2026 年 4 月。兩個月前，這個數字是 160 萬。翻倍，用了不到 60 天。

同一週，Anthropic 在沒有太多預告的情況下，一口氣推出了兩個新產品方向，並且更新了 Claude Code 的核心設定架構。技術圈在消化 GPT-5.5 的同時，後面還有一堆東西在安靜地落地。

Claude Code 2.1.119 — config 設定終於持久化了

這個更新聽起來不大，但對每天用 Claude Code 的開發者來說，它解決了一個每次新開 session 都要重設的痛點。

從 2.1.119 開始，/config 指令的設定（主題、editor mode、verbose 輸出等）會自動寫入 ~/.claude/settings.json，並且參與 project/local/policy 的 override 優先序。換句話說，你設定一次，下次開 Claude Code 就是你的環境，不用每次重來。

同一版本還加了 prUrlTemplate 設定，讓你把 PR URL 的格式綁在 project 設定裡；MCP 的支援也進一步加強，hook 機制的權限邏輯更細緻。

這是一個把 Claude Code 往「真正的日常工具」而不是「每次要重新調校的工具」推的方向。小更新，但累積效益是你每天少花幾分鐘在環境設定，長期下來不小。

每一個讓工具「消失」的改動——讓你不需要去想它怎麼設置——都比新增一個亮眼功能更難做。設定持久化是基礎建設，不是行銷點，但正因為如此，它比行銷點更值得記。

Anthropic 推出 Claude Design — AI 生成視覺的第一個正式產品

4 月 17 日，Anthropic 正式推出 Claude Design，底層用的是 Claude Opus 4.7。

功能定位：讓使用者直接用對話生成視覺稿——prototype、slide deck、one-pager、信件模板。目前是 research preview 狀態，開放給 Pro、Max、Team、Enterprise 訂閱者。

這件事有兩層要看。第一層是產品：Claude 在文字生成、code 生成之後，正在系統性地往視覺生成擴張，而且不是直接競爭 Midjourney 或 DALL·E 的圖像生成，是更偏 UI/文件的實用視覺。第二層是策略：Anthropic 一直以來的定位是「API 優先、B2B 優先」，Claude Design 是第一個明確面向非技術用戶、主打直覺操作的消費者產品。

「我們幫你找漏洞」（Glasswing）加上「我們幫你做設計」（Claude Design），加上「我們幫你跑長時間任務」（Managed Agents）——Anthropic 正在從單一 API 廠商走向全棧工具公司。問題是這個節奏能維持多久，以及每個新方向的深度夠不夠讓用戶留下來。

原文來源：Anthropic launches Claude Design - TechCrunch

Claude Managed Agents — 長時間任務的新基礎設施

同樣是這波更新，Anthropic 推出了 Managed Agents，定位是 Claude Platform 上的 hosted service，專門處理「長時間、多步驟的 agent 工作流」。

技術特點：stable session 介面、built-in sandbox、durable state（跨工具呼叫保持狀態）、更快的 agent 啟動速度。

這是在填一個真實的痛點。現在跑 AI agent 工作流的工程師，最常踩的坑是：session 斷掉、中間狀態不見、工具呼叫失敗後要從頭來過。Managed Agents 的設計意圖是讓這些事情在基礎設施層處理掉，讓上層的邏輯更乾淨。

系統性的含義：Anthropic 在推動一個觀點——AI agent 需要跟傳統微服務一樣的基礎設施保障（持久狀態、可靠重試、隔離沙盒）。這個觀點如果被市場接受，下一步就是整個 agent 工作流的設計範式要跟著改。

OpenAI Codex: 300 萬週活躍開發者，兩個月翻倍

OpenAI 公布，截至 2026 年 4 月，Codex 的週活躍開發者突破 300 萬。兩個月前（2026 年 3 月初）是 160 萬。

這個增速跟 Claude Code 的 ARR 在同一個量級的故事（$10 億，六個月達到），都在說同一件事：AI coding agent 這個類別正在高速從「早期採用者玩具」走向「日常工具」。

逆向問：這 300 萬開發者裡，有多少是在用 OpenAI 官方的 Codex 環境，有多少是在用 Codex API 但介面是 Cursor / Windsurf / 其他？數字是總量，但工具入口的分散化意味著 OpenAI 和這些工具廠商之間有一個複雜的競合關係在發展。

誰控制使用者介面，就控制使用者習慣。OpenAI 現在在想的問題，跟 AWS 當年想「要不要做自己的服務還是只提供雲端算力」是類似的結構。

原文來源：OpenAI Codex Release Notes - April 2026 | Releasebot

Windsurf 引入 Devin Cloud — 本地任務委派到雲端

Windsurf 最新版本（v2.0.61，4 月 17 日）加入了 Devin Cloud 整合——讓使用者在本地 session 中把任務委派給雲端 agent 執行。

具體機制：你在本地的 Windsurf 環境裡跑到一半，可以把某個複雜的長時間任務「推」到雲端跑，費用從你現有的 quota 扣，額外用量另計。同一版本還加入了 Adaptive model，自動在不同任務間選擇最合適的模型（平衡品質、延遲、成本），讓 quota 撐更久。

「本地工具 + 雲端算力」的組合，是幾家 AI coding 工具都在收斂的方向——Claude Code 有 remote session、OpenAI Codex 有雲端環境、現在 Windsurf 也有 Devin Cloud。

有意思的地方是：哪種模式最後「贏」，不一定是技術上最強的，而是最符合開發者心智模型的。本地工具感覺有掌控感，雲端有彈性——把兩個結合的工具，在使用者體驗上需要解決「我到底在哪裡跑？結果在哪裡？」的定向問題。

原文來源：Windsurf Changelog - April 2026 | Releasebot

這週的幾條新聞有一個共同的結構：工具層正在往基礎設施層長。config 持久化、managed agent session、雲端委派——這些不是功能點，是工具把自己從「偶爾用的輔助工具」變成「不得不依賴的基礎設施」的過程。

基礎設施級的黏性比功能黏性高一個量級。一旦你的工作流依賴某個工具的 session 狀態，換工具的代價就不只是學習曲線，是整個流程要重建。這是所有工具廠商現在在搶的位置——不是要做最聰明的 AI，是要做你工作流裡最難被換掉的那個。
