上班第一件事不是工作，而是拼湊今天要做什麼 — 我把它換成一封自動寄來的信

發表於2026-07-01|更新於2026-07-01|Claude Code
|總字數:1.9k|閱讀時間:5分鐘|瀏覽量:

打開電腦，第一件事不是工作。是打開四個分頁，拼湊出「今天到底要做什麼」。

行事曆看一遍，哪幾個會議、幾點、跟誰。信箱掃一遍，昨晚有沒有人丟東西進來、哪封是急的。專案看板點開，我手上那幾張卡現在卡在哪。然後在腦袋裡把這三份東西疊起來，排出一個順序。等這套動作做完，通常已經過了十分鐘，而我一行 code 都還沒寫。

這十分鐘每天都在發生。它不難，但它很煩——煩的地方在於它是純搬運：把散在三個地方的資訊集中到一個地方，再做一次判斷。搬運這件事，本來就不該由你來做。

舊做法：你是那個把資料撈出來的人

先看清楚舊做法到底在做什麼，才知道哪裡可以換掉。

早上這十分鐘，你其實同時在扮演兩個角色。一個是「資料撈取員」——去日曆、信箱、看板各撈一份出來。另一個是「判斷者」——把三份資料合起來，決定今天先做哪個、哪個有風險、哪個可以晚點。

問題是，這兩個角色的價值天差地遠。撈資料是機械動作，換誰做結果都一樣；判斷才是只有你能做的事，因為只有你知道這週的重點、哪個客戶不能拖、哪個會議其實可以掛著聽。

舊做法把你 90% 的時間花在那個換誰做都一樣的角色上。你不是在早上做決策，你是在早上做資料庫查詢，查完才輪到決策。這就是那十分鐘真正的浪費——不是它花了時間，是它把你最不值錢的動作排在一天的最前面。

新做法：讓一封信在你到公司前就寫好

現在同一件事變成這樣：我到公司、打開電腦，信箱裡已經躺著一封信，主旨是「每日工作摘要 — 今天的日期」。點開，今天的會議按時間排好、未讀信裡哪幾封需要回、看板上進行中的卡整理成一份清單，最上面還有一段「今天最該先處理的三件事」。

我要做的只剩一件：讀完,然後開始做。撈資料的那個角色被裁掉了，我直接從判斷開始。

這封信不是我寫的，是一個掛在 Claude Code 上的技能寫的。拆開來看，它就是把我早上那套動作原封不動搬給機器跑，只是機器不會覺得煩：

第一步，並行撈三份資料。 日曆走 Google Calendar 的 MCP 連接器，只抓今天、只看公司網域的與會者，順便偵測時間有沒有撞車;信箱走 Gmail MCP，用 is:unread 撈未讀,標出看起來急的；工作單走 Notion，把狀態是「進行中」「測試中」的卡撈出來。這三件事沒有先後關係，同時發。

第二步，做判斷。 這步是整個技能的重點,也是唯一機器在「動腦」的地方——把三份資料合起來,排出優先順序、標出風險、給出建議動作。前面兩步任何腳本都做得到,值錢的是這步。

第三步,包成一封人看得下去的信。 產出 HTML,寄出去。

魔鬼藏在「寄信」這一步

如果你以為難的是串接那些 API，那就搞反了。串接反而是最直的一段。真正踩坑的地方，是「把信寄出去、而且長得好看」。

先講寄。最省事的做法是 SMTP 直送:拿一組 Gmail 應用程式密碼（注意,是應用程式密碼那組 16 碼,不是你的登入密碼),塞進 ~/.claude/settings.json 的 env 區塊,一段 smtplib 就能把信送出去。萬一哪天 SMTP 連不上,退一步用 Gmail MCP 建草稿,至少信還在,你手動按送出就好。留一條退路,別讓整套流程卡死在最後一哩。

再講好看,這裡有個反直覺的坑:Gmail 會把你 <head> 裡的 CSS 和 <style> 標籤整段刪掉。你辛苦寫的樣式,在收件匣裡會全部失效。

這件事第一次遇到會很困惑——本機打開明明好好的,一寄出去就變成裸 HTML。原因是信箱這個環境跟瀏覽器不一樣,它不信任你的 <style>。解法只有一個:所有樣式全部寫成 inline style,直接掛在每個 <div> 上。這不是「比較保險」,是唯一會動的寫法。

打個比方,寫網頁像在自己家裡裝潢,水電怎麼牽都行;寫信件 HTML 像在租來的套房,房東（信箱）早就把牆封死了,你只能用貼的。認清楚你在誰的地盤上,少走很多冤枉路。

最後一哩:讓它在你睡覺時就跑完

到這裡,技能已經能動了——但你還得自己去按它。真正把「早上十分鐘」歸零的,是排程。

Claude Code 的 /schedule 可以把這個技能掛成一個雲端 routine,設成每個工作日早上八點半跑。重點是「雲端」兩個字:它不是在你的筆電上跑,是在遠端跑,所以你電腦關著、還在睡,它照樣把信寄出來。等你起床,信已經在那了。

雲端執行有個必須知道的邊界:遠端那台機器碰不到你本機的檔案、環境變數、本機 MCP。所以本機版靠 settings.json 的憑證那套,到雲端全部失效。解法是把需要的東西改成雲端拿得到的形式——日曆、信箱走 claude.ai 上已經授權的雲端連接器;Notion 沒有雲端連接器,就直接用 REST API 打;寄信的憑證,內嵌進排程的 prompt 裡。同一個技能,本機跟雲端要餵它不一樣的養分,這點不搞清楚,排程會安靜地失敗給你看。

還有個小陷阱是時區。cron 寫 30 0 * * 1-5 看起來像凌晨,其實那是 UTC,換算台北剛好是早上八點半。排程沒在你以為的時間跑,先去看是不是時區的鍋。

真正被换掉的,不是那十分鐘

把技能拆到這裡,你會發現它省下的其實不是時間。十分鐘而已,一天省十分鐘不會改變你的人生。

它換掉的是順序。以前你一天的開頭,是從一堆原始資料裡自己拼出一張圖;現在你的開頭,是直接拿到那張圖,從判斷開始。你沒有變得更快,你是把最不值錢的那段動作,從一天的最前面整個拿掉了。

這也是自動化最容易被誤解的地方。大部分人以為自動化是「讓機器幫我做事」,所以拼命找那些花很多時間的大任務去自動化。但真正值得先動手的,往往是這種小到不好意思講、卻天天發生、而且純搬運的動作。它不花時間,它偷的是你一天裡最清醒的那段注意力。

所以下次想自動化什麼,先別問「哪件事最花時間」,問「哪件事換誰做結果都一樣,卻硬是排在我腦子最好用的時候」。那件事,才是你該寄給自己的第一封信。
